  <action>
**ado-client.mjs additions** — Add two new exported functions at the end (before any closing comments), following the exact error handling pattern of existing functions (buildAuthHeader, 203 check, typed error objects):

1. `adoGetTeamIterations(config, teamId, timeframe = null)` — GET `{orgUrl}/{project}/{teamId}/_apis/work/teamsettings/iterations?api-version=7.1`. If timeframe provided, add `$timeframe` query param. Returns `{ value: [...iterations] }`.

2. `adoGetIterationWorkItems(config, teamId, iterationId)` — GET `{orgUrl}/{project}/{teamId}/_apis/work/teamsettings/iterations/{iterationId}/workitems?api-version=7.1`. Returns `{ workItemRelations: [...] }`.

Both functions: use `buildAuthHeader(config.pat)`, handle 401/203 as auth error, 403 as permission error, non-ok as api error, catch block for network error. Same pattern as `adoGetTeamMembers`.

**sprint.mjs** — New script following bugs.mjs pattern exactly:

Arg parsing: `--sprints=N` (default 3, velocity trend depth), `--check-config`, `--days` (unused but accepted for summary.mjs passthrough compatibility).

Main flow:
1. Load config, get project info via `adoGetProject` to obtain `defaultTeam.id`
2. Fetch all iterations via `adoGetTeamIterations(config, teamId)` — no timeframe filter (need all for velocity)
3. If `value` is empty array: output `{ error: { type: 'no_sprints', message: 'No sprints configured for this project. /adi:sprint requires Azure DevOps iterations.' } }` and exit
4. Classify iterations by `attributes.timeFrame`: 'current', 'past', 'future'
5. Current sprint processing:
   - Get iteration work items via `adoGetIterationWorkItems`, extract unique target IDs, batch-fetch with fields: `System.Id, System.Title, System.State, System.WorkItemType, System.AssignedTo, System.CreatedDate, Microsoft.VSTS.Scheduling.StoryPoints, Microsoft.VSTS.Scheduling.Effort, Microsoft.VSTS.Scheduling.RemainingWork`
   - Compute completion: classify each item state against done states `['Done','Closed','Completed','Resolved']`. Count/sum done, in-progress (everything else non-new), not-started (New, To Do, or null state)
   - Story points: prefer `StoryPoints`, fall back to `Effort`, fall back to item count (set `useItemCount: true` flag in output)
   - Backlog health: count unestimated (no StoryPoints AND no Effort), unassigned (no AssignedTo), scope creep (items with CreatedDate after sprint startDate)
   - Burndown: use heuristic from RESEARCH — elapsed time % vs completed work %, 10% margin for on-track, 25% for at-risk, else behind. If sprint has no dates, status = 'unknown'
6. Velocity tracking:
   - Take past iterations sorted by finishDate descending, slice to `--sprints=N`
   - For each: fetch work items, sum completed points (same done-state check)
   - Output array of `{ name, completedPoints, totalPoints, itemCount }`
7. Output JSON to stdout: `{ sprint: { name, startDate, finishDate, completion: { done, inProgress, notStarted, totalItems, donePoints, totalPoints, useItemCount }, backlogHealth: { unestimated, unassigned, scopeCreep }, burndown: { status, elapsedPct, completedPct } }, velocity: [...], meta: { project, generatedAt, sprintsAnalyzed } }`

Progress messages to stderr only. JSON-only stdout.
